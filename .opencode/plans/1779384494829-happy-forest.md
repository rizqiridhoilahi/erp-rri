# Fix for DeepSeek Thinking Mode Error in Opencode

## Problem
User experiences HTTP 400 error when using opencode CLI:
```
RequestExecutor.execute: Provider request failed with HTTP 400: {"error":{"message":"Error from provider (DeepSeek): The `reasoning_content` in the thinking mode must be passed back to the API.","type":"invalid_request_error","param":null,"code":"invalid_request_error"}}
```

This occurs when:
- Using models that route through DeepSeek infrastructure (including opencode free tier "big pickle" model)
- Thinking mode is enabled (either by default or explicitly)
- Multi-turn conversations or tool calls are involved

## Root Cause
DeepSeek's thinking mode API requires ALL assistant messages in conversation history to include the `reasoning_content` field (even empty string ""). Opencode's current implementation strips this field when rebuilding message history for subsequent turns, causing DeepSeek to reject the request on the second+ turn.

## Solution
Add `interleaved` configuration to the model settings in `opencode.json` to properly handle the `reasoning_content` field as required by DeepSeek's API.

## Files to Modify
- `/home/ubuntu/.config/opencode/opencode.json`

## Changes Required
In the `providers.deepseek.models.deepseek-v4-flash` section, add:
```json
"interleaved": {
  "field": "reasoning_content"
}
```

## Verification Steps
1. Restart opencode CLI after making the change
2. Start a new conversation
3. Send multiple messages or trigger tool usage
4. Verify no HTTP 400 error occurs
5. Confirm the model works correctly in multi-turn scenarios

## Notes
- The same fix should be applied to `deepseek-v4-pro` if used
- This fix addresses the core issue without disabling thinking mode, preserving the model's reasoning capabilities
- If the issue persists, consider disabling thinking mode temporarily by setting `"thinking": { "type": "disabled" }` in model options
